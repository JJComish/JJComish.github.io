Motion Estimation on Embedded System

My partner Marcin Leciej and I were tasked with writting a program for an embedded system that used the sum of absolute differences algorithm for calculating the vectors when attempting motion estimation between two images. The goal of the project was to write the most efficient implementation that would then be run on a 32 bit ARM processor. The project was done in three stages. An initial naive implementation was written in pure C where the only optimization was using cache efficient array traversal. The second iteration of the program used Neon Intrinsics and software pipelining to reduce the number of cycles required to compute the SAD values. The final iteration invovled assembly inlining that would be used if the a new hardware supported instruction could be implemented. Alongside this code was the design of the theoretical hardware instruction.

Neptune Banking

Neptune Banking was a mock legacy banking application that used a Spring Boot (Java) back-end and Node Js front-end. It suffered from a series of bugs, code quality and security issues, and missing features. Working in a team of 6 a requirements specification document was written to match provided version of the application along with a list of desired features. A continous integration pipleine was implemented using Jenkins that, following a peer code review of merge requests, performed unit tests, front-end tests with Selenium, and scanned the application for code quality and security issues using SonarQube. Over the course of 4 months bugs were fixed, features were implemented, and unit tests and front-end tests were written. Performance testing was done using Apache Jmeter with Perfmon. Security and penetration testing was conducted using OWasp Zap.

SimpleStockShare

SimpleStockShare is a mock stock trading application that is designed with software scalability in mind. Originally started in a team of three but maintained after one of the developers stopped contributing and then dropped out of the class. The application was broken into a series of dockerized components to allow for distribution across resources. The application had strict requirements including the use of a preexisting server that supplied the quoted values of stocks. The main transaction server componenet used Spring Boot. Quotes from the legacy quote server were stored, where applicable, in a cache implemented using Redis. The application used Glowroot for performance analysis and a workload generator written in Python designed to simulate n users simultaneously making transacion requests. The final implementation of the application easily supported 1200 concurrent users with an average transaciton speed of ~25,000 transactions per minute.
